本页只聚焦“回放”链路，基于 2026 年 4 月 3 日仓库当前代码进行静态分析。从主窗口点击“开始回放”开始，沿着场景选择、trace 装载、统一时间轴调度、信号覆盖、帧批处理，到 ZLG 设备上的 CAN FD 发送路径，梳理真实执行流程和关键分支。

范围：回放，不含录制 重点：CAN FD 实际发送链路 验证方式：源码阅读 + 现有测试对照

先看结论

当前实现里，CAN FD 回放和 CAN/J1939 共用同一个统一时间轴，但在“批次是否交给设备定时队列”这一步明显分叉。

  • ReplayApplication.start_replay() 会加载 trace、按 ts_ns 排序、加载数据库、注入初始信号覆盖、创建适配器，然后把所有内容交给 ReplayEngine。src/replay_platform/app_controller.py:188-218
  • ReplayEngine 以 ScenarioSpec.timeline_items() 构造统一时间轴，把帧事件、链路动作、诊断动作混排。src/replay_platform/core.py:352-364
  • 帧批处理窗口固定为 2 ms。一个窗口内连续的帧会先聚成批，再决定是“立即发”还是“设备定时发”。src/replay_platform/runtime/engine.py:38-41 / 367-386

总流程图

1
UI 决定启动来源

优先使用场景绑定的 trace_file_ids；若为空且主窗口选中了 trace，则回退到“当前选中文件”。

2
应用层装载数据

读取 trace 缓存，转成 FrameEvent，并装载数据库绑定、初始信号覆盖、适配器和诊断对象。

3
引擎构造统一时间轴

把帧、链路动作、诊断动作统一按 ts_ns 排序，形成单一调度序列。

4
启动设备通道

按绑定配置打开适配器、启动物理通道；CAN FD 通道会下发名义波特率和数据相位波特率。

5
2ms 分片调度

连续帧先做 2 ms 切片，再过滤帧使能、应用信号覆盖、逻辑通道映射到物理通道。

6
ZLG 发出 CAN FD

按物理通道分组进入 TransmitFD；真实 payload 长度写入设备帧长度，BRS 由 flags 控制。

当前实现最关键的 4 个判断点

  1. 启动时文件来源是否来自场景，还是来自 UI 回退选择。
  2. 2 ms 切片内是否有至少 2 条“仍然启用”的帧。
  3. 该批次是否全部属于 CAN/J1939 且适配器支持 queue_send。
  4. 链路动作是否插入在帧之间，导致批次被打断，或者因为重连耗时让后续帧变成“已过期后立即发送”。
对 CAN FD 的直接含义 当前代码里，只要批次里存在 CAN FD 帧，就不会走引擎的“设备定时队列”分支；这意味着引擎不会为 CAN FD 显式保留 slice 内的微小时隙。

详细流程拆解

1. 主窗口决定“这次回放到底放哪些文件”

主窗口的 _scenario_from_current_source() 会先取当前场景；如果场景本身没有 trace_file_ids，而主窗口左侧列表选中了 trace，则把这些 trace 临时注入到 payload 中，并把启动来源标成 SELECTED_FALLBACK。随后 _start_replay() 把这个场景连同“循环回放”开关一起交给应用层。

src/replay_platform/ui/main_window.py:3510-3538

2. 应用层把 trace 变成运行期帧列表

ReplayApplication.start_replay() 会遍历 scenario.trace_file_ids，通过文件库读取缓存后的 trace 事件，并把 source_file 改成原始文件路径或文件名。随后所有帧按 ts_ns 再排一次序，保证多文件合并后仍是统一时间轴。

src/replay_platform/app_controller.py:188-203

3. 启动前先把“发送前改包”能力装进来

数据库绑定会先调用 load_database()，随后场景自带的 signal_overrides 逐条注入。真正发帧前，引擎会在 _prepare_frame_groups() 里调用 signal_overrides.apply()，因此覆盖发生在“逻辑通道帧”转成“物理通道帧”之前。

src/replay_platform/app_controller.py:204-207 src/replay_platform/runtime/engine.py:428-446 src/replay_platform/services/signal_catalog.py:192-208

4. 引擎把帧、链路动作、诊断动作合成一条时间轴

ScenarioSpec.timeline_items() 直接把三类对象放进一个列表，再按 ts_ns 排序。这意味着回放调度看到的是“统一时间轴”，而不是“帧一条线、链路动作一条线”。所以链路动作能在时间轴中间打断一个帧批次。

src/replay_platform/core.py:352-364

5. 引擎启动时建立逻辑时间基准并打开物理通道

ReplayEngine.start() 会先 _prepare_channels()，再以当前 time.perf_counter_ns() 建立 _base_perf_ns。对于 ZLG，若通道绑定类型是 CAN FD，则 start_channel() 会把通道初始化为 ZCAN_TYPE_CANFD，并下发 canfd_abit_baud_rate 和 canfd_dbit_baud_rate。

src/replay_platform/runtime/engine.py:122-138 src/replay_platform/runtime/engine.py:205-230 src/replay_platform/adapters/zlg.py:89-105 / 258-263

6. 运行循环按“当前索引 + 2ms 分片”拿出待发帧

_run_loop() 每次先看当前位置是不是帧；如果是帧，则调用 _frame_batch_at()，把从当前帧开始、到 2 ms 窗口结束前的连续帧打成一个批次。只要中间插入了诊断动作或链路动作，批次就会被截断。

src/replay_platform/runtime/engine.py:251-289 src/replay_platform/runtime/engine.py:367-384

7. 只有满足条件时才走“设备定时发送”

_should_schedule_frame_batch() 的判断很严格：必须至少有 2 条仍然启用的帧，第一条已经到点，最后一条还在 2 ms 调度窗口内，且每条帧的总线类型都属于 CAN/J1939，同时适配器声明支持 queue_send。因为 SCHEDULED_FRAME_BUS_TYPES 不包含 CAN FD，所以 CAN FD 天然走不到这个分支。

src/replay_platform/runtime/engine.py:38-41 src/replay_platform/runtime/engine.py:386-416

8. 帧真正发送前还会经过两层处理

第一层是 FrameEnableService，按“逻辑通道 + 报文 ID”判断这帧是不是被临时禁用；第二层是 SignalOverrideService.apply()，按 DBC 对 payload 做重新编码。对于 CAN FD，如果改包后 payload 长度变化，会再次通过 canfd_payload_length_to_dlc() 把真实长度映射回内部 DLC 码值。

src/replay_platform/services/frame_enable.py:8-34 src/replay_platform/runtime/engine.py:421-446 src/replay_platform/services/signal_catalog.py:205-208

9. 逻辑通道在这里才映射成真实设备通道

引擎按场景里的 binding 找到逻辑通道对应的适配器和物理通道，然后把帧 clone 成“物理通道版本”，再按适配器分组。这个阶段之后，同一个适配器内的原始顺序会被保留，但已经不再区分回放文件来源。

src/replay_platform/runtime/engine.py:428-446

10. ZLG CAN FD 发送使用真实 payload 长度，而不是内部 DLC 码值

ZLG 适配器会先按物理通道再拆一次。对于 CAN FD，_send_fd() 会把 payload 截到 64 字节以内，然后把 len(payload) 写入 SDK 结构的 frame.len。BRS 通过 flags & 0x01 写回，tx_echo 通过 0x20 写回；但从当前实现看，导入时保留的 esi 并没有在发送时参与写入。

CAN FD 的特殊路径

环节 当前逻辑 对回放行为的影响
Trace 导入 ASC 会解析 brs/esi/dlc/data_length；BLF 会根据 is_fd 判定 CAN FD，并把 payload 长度重新映射为 DLC 码值。 内部统一成 FrameEvent(bus_type=CANFD)，后续不再依赖源文件格式。
统一时间轴调度 和 CAN/J1939 共用同一个 ts_ns 逻辑时间轴。 暂停 / 恢复 / 循环回放对 CAN FD 同样生效。
2ms 分片 CAN FD 也会进入 2 ms 分片，但不会进入设备定时队列判断的成功分支。 同一 slice 内多个 CAN FD 帧会被立刻批量下发，而不是由引擎显式保留微小时隙。
信号覆盖 如 DBC 重编码后字节数变化，会重算 DLC 码值。 内部 DLC 一致性可以维持，但最终下发给 ZLG 的仍是“真实字节长度”。
ZLG 发送 TransmitFD 的 frame.len 直接使用 payload 字节数。 16/24/64 字节这类 CAN FD 负载会按真实长度下发，不靠 DLC 码值反解。

链路动作对回放的影响

  • 链路动作是时间轴里的正式事件，不是旁路逻辑。src/replay_platform/core.py:261
  • 引擎遇到 LinkAction 时会同步执行断开或重连；如果指定了逻辑通道，还会重新 start_channel()。src/replay_platform/runtime/engine.py:489-507
  • 由于链路动作会打断连续帧窗口，所以一个本来能合批的 slice 会被拆开。现有测试已经覆盖这个行为。tests/test_engine.py:597 / 1077

我认为最值得关注的实现特征

  1. 当前 CAN FD 回放更像“统一时间轴 + 立即批量发送”，而不是“设备侧精确定时发送”。2 ms slice 仍然存在，但更多是为了统一调度和合批，不是为了给 CAN FD 保留精细间隔。
  2. 信号覆盖在发送前最后一跳生效，因此 CAN FD payload 长度理论上可以被 DBC 重编码改变；引擎会同步修正内部 DLC。
  3. 链路动作是强同步点，它会打断合批，甚至可能把后续帧推成“超时后立即发送”的状态。
  4. 暂停/继续处理得比较干净，它通过平移基准时间避免了暂停期间的时间追赶，这一点对 CAN FD 也成立。
