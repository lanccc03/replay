同星 / ZLG 设备在 CAN / CANFD 下的回放调度策略分析

基于当前仓库源码与单元测试 分析日期：2026-04-09 范围：回放调度，不含原始 ETH 回放 边界：未做 Windows 真机硬件联调

这份报告把回放策略拆成两层来看：第一层是 统一运行时调度，也就是 ReplayEngine 如何在同一条时间轴上调度帧、诊断和链路动作；第二层是 设备适配器发送策略，也就是 ZLG / 同星各自如何把到期帧真正下发到 CAN / CANFD 总线。

结论先行：ZLG 仅对经典 CAN/J1939 提供“设备排队发送”路径，CAN FD 会退回即时发送；同星在当前实现里不声明设备排队发送能力，因此 CAN 和 CAN FD 都走“引擎等待到点 + 异步立即发”路线。

ZLG / CAN

支持 2ms 分片内的设备排队发送，能把同一批里的剩余间隔编码成 queue_delay_us 下发给 ZLG。

ZLG / CANFD

虽然适配器支持 CAN FD 收发，但当前调度路径明确不对 CAN FD 走 send_scheduled，仍按到点即时发送。

同星 / CAN

引擎不会把它当作可排队设备；到达目标时刻后，逐帧调用 tsapp_transmit_can_async。

同星 / CANFD

与 CAN 同策略，只是发包 API 换成 tsapp_transmit_canfd_async；关闭时会尽量等待 TX FIFO 清空。

统一调度主线

统一策略由运行时层决定，设备只是在最后一步决定“是硬件排队发送，还是立即发”。核心入口包括： src/replay_platform/app_controller.py:275-320、 src/replay_platform/core.py:370-374、 src/replay_platform/runtime/engine.py:38-41、 src/replay_platform/runtime/engine.py:285-324、 src/replay_platform/runtime/engine.py:420-505。

1. 载入与归并

App 层先把 trace 帧加载出来，必要时按绑定把源通道映射到逻辑通道，然后按 ts_ns 全局排序。

2. 统一时间轴

ScenarioSpec.timeline_items() 把帧、诊断动作、链路动作放进同一时间轴并按 ts_ns 排序。

3. 2ms 分片

运行时把连续帧切成最多 2ms 的小片段。碰到诊断或链路动作，批次立即截断，不跨越非帧事件。

4. 调度判定

只有“至少 2 帧、第一帧已到点、最后一帧仍在未来 2ms 内、所有帧都属于可排队总线、所有相关适配器都声明 queue_send”时，才走设备排队发送。

5. 真正下发

满足条件就调用适配器 send_scheduled()，否则等到目标时刻后直接 send()。诊断则由独立线程异步执行。

暂停 / 恢复：引擎在恢复时把 _base_perf_ns 平移到新的性能计时基准上，因此逻辑时间轴不重排，只是整体后移。 这意味着 ZLG 的经典 CAN 排队路径和同星的立即发送路径，都会共享同一套暂停/恢复基准。

ZLG 详细拆解

  • 适配器能力：src/replay_platform/adapters/zlg.py:194-204 把 queue_send 声明为 True，因此运行时允许它进入设备排队路径。
  • 经典 CAN / J1939：src/replay_platform/adapters/zlg.py:133-143 会根据 enqueue_base_ns + item.ts_ns - now_ns 计算剩余延时，并写入 queue_delay_us。
  • 延时编码方式：src/replay_platform/adapters/zlg.py:286-296 把延时拆进 _pad / _res0 / _res1 后，再走 Transmit()。
  • CAN FD：src/replay_platform/adapters/zlg.py:134-135 发现批次里有 CAN FD 就直接退回普通 send()；_send_fd() 通过 TransmitFD() 即时发出。
  • 报文长度：src/replay_platform/adapters/zlg.py:305-313 明确使用“真实字节数”填 len，不是把内部 DLC 码值原样塞给设备。
含义：当前实现里，ZLG 的“精细保时”能力只覆盖经典 CAN/J1939；CAN FD 虽然能发，但不走同样的硬件队列调度。

同星详细拆解

  • 适配器能力：src/replay_platform/adapters/tongxing.py:432-438 只声明 can / canfd / j1939 / can_uds，没有 queue_send。
  • 发送主线：src/replay_platform/adapters/tongxing.py:371-388 在运行时锁内逐帧发送；成功就累计，失败时若前面已有部分成功则直接返回已发数。
  • CAN 与 CANFD：src/replay_platform/adapters/tongxing.py:580-587 分别使用 tsapp_transmit_can_async() 与 tsapp_transmit_canfd_async()。
  • 关闭时清尾：src/replay_platform/adapters/tongxing.py:305-320 与 589-618 会在 close 前轮询 TX FIFO，尽量等设备侧待发帧清空。
  • 不能从当前代码推断为“设备未来时刻排队发送”：虽然 CAN/CANFD 结构体里都写入了 FTimeUs，但仓库里没有把同星适配器接入 queue_send 判定，因此运行时不会把它当作硬件排队设备。
含义：同星当前更像“到点触发异步发送”的执行器，而不是“先排入设备队列、由设备保留批内时间差”的执行器。

运行时层的几个关键细节

  • 时间轴归并：src/replay_platform/core.py:370-374 统一把帧、诊断、链路动作按 ts_ns 排序。由 Python 稳定排序可推断：同一时间戳下，原先先入列表的 frame 会先于 diagnostic / link 排序保留，这一点是基于源码行为的推断，不是显式注释承诺。
  • 2ms 分片：src/replay_platform/runtime/engine.py:38-41 与 401-418 规定批次窗口为 2,000,000ns；只要中间出现非帧事件，批次就被切断。
  • 排队发送条件很严格：src/replay_platform/runtime/engine.py:420-450 要求至少两帧、第一帧已经到点、最后一帧仍在未来 2ms 内、所有帧都属于 CAN/J1939，且所有涉及适配器都支持 queue_send。
  • 信号启停会影响调度判定：src/replay_platform/runtime/engine.py:425-427 先按 FrameEnableService 过滤。如果批里原本有两帧，但只剩一帧仍启用，就直接退回即时发送。
  • 诊断不会阻塞后续帧：诊断动作单独进 _diagnostic_thread，因此慢诊断不会卡住后面已经到点的帧发送。
  • 链路动作是调度屏障：因为 _frame_batch_at() 只吸收连续帧，所以断连/重连动作会天然阻止前后帧合并成一个批次。

对实际回放效果的含义

  • 如果你最关心经典 CAN 的批内间隔保持：当前实现里 ZLG 更强，因为它能把 2ms 窗口内的剩余时间差交给设备队列。
  • 如果你最关心 CAN FD 的时间精度：当前两条路径都主要依赖引擎线程在用户态“等到点再发”，没有统一的设备排队发送机制。
  • 如果你希望 ZLG 与同星行为一致：要么给同星补 queue_send/send_scheduled 路径，要么把 ZLG 的经典 CAN 也改回即时发送，换取跨设备一致性。
  • 如果你担心诊断动作影响总线回放：当前实现已经把诊断分到独立线程，慢诊断不会直接卡住后续帧，但时间轴上的诊断动作本身仍会作为“非帧事件”打断批次。

结论摘要

当前仓库的回放调度策略可以概括成一句话： 统一时间轴由 ReplayEngine 掌控，ZLG 只在经典 CAN/J1939 上承接 2ms 窗口内的设备排队发送，而同星在 CAN / CANFD 下都停留在“引擎等到时刻后调用异步发送 API”的层级。

因此，ZLG 与同星在 CANFD 下的策略实际上更接近，都是即时发送；真正拉开差距的是 经典 CAN， 尤其是存在 2ms 内多帧密集发送时，ZLG 的硬件队列路径更有机会保住批内间隔。
