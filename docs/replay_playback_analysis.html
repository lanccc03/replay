Current Implementation

当前回放的核心不是“顺序播 trace”，而是“以逻辑时间轴驱动帧与链路动作”。

现在这套实现由 ReplayApplication 负责启动前装配，由 ReplayEngine 负责运行期调度。调度里面最关键的几个点是：多 trace 合并排序、统一时间轴、2ms 帧分片、20ms 计划发送观察窗、 帧使能过滤、发送前信号覆盖、逻辑通道到物理通道映射、暂停恢复时钟平移，以及循环回放时的通道重初始化。

2ms 连续帧分片窗口，来自 FRAME_SLICE_WINDOW_NS
20ms 计划发送预排队窗口，来自 FRAME_QUEUE_LOOKAHEAD_NS
3段 UI 触发、应用装配、引擎调度三段式主路径
2类 本文只展开 FrameEvent 与 LinkAction
本页范围
  • 保留链路动作，因为它是时间轴屏障，会直接改变调度行为。
  • 不展开诊断队列、UDS/DoIP 和诊断响应细节。
  • 所有结论都基于当前仓库实现与现有测试，而不是规划中的理想模型。

启动主链路

从主窗口点击“开始回放”到后台线程真正进入调度循环，当前路径大致分四段。

入口链路
1. UI 先确定“本次用哪个场景、哪个 trace 来源启动”
main_window.py:3510 的 _scenario_from_current_source() 会先取当前场景。 如果场景自己的 trace_file_ids 为空，但主窗口左侧已经选中了 trace 文件， 则自动回退到“主窗口选中文件”启动，并把 launch_source 设为 ReplayLaunchSource.SELECTED_FALLBACK。
2. 应用层装配运行输入
app_controller.py:188 的 start_replay() 会先清空运行态帧禁用规则， 然后逐个读取场景绑定的 trace；把所有帧合并到一个列表里，并在 app_controller.py:203 按 ts_ns 统一排序。
3. 启动前加载发送侧辅助能力
对每个数据库绑定执行 signal_overrides.load_database()， 对场景中的每条信号覆盖执行 signal_overrides.set_override()， 再构造适配器实例并把它们交给引擎。
4. 引擎配置并启动后台线程
engine.configure() 负责生成统一时间轴与初始快照， engine.start() 负责打开适配器、启动物理通道、设置 _base_perf_ns，然后启动 replay-engine 线程进入 _run_loop()。
启动来源的实际意义 “场景绑定文件”和“主窗口选中文件回退”最终都会走同一套回放引擎；它们的差异只在于启动前喂给引擎的 trace 集合不同。

统一时间轴模型

当前调度依赖统一时间轴，而不是“trace 线程”和“动作线程”各自跑。

数据契约
对象 作用 调度关键字段 本文关注点
FrameEvent 总线帧事件 ts_ns、bus_type、channel、message_id、payload 分片、计划发送、通道映射、信号覆盖
LinkAction 链路断开/恢复事件 ts_ns、adapter_id、logical_channel、action 时间轴屏障、通道状态切换
ReplayRuntimeSnapshot 运行态快照 current_ts_ns、timeline_index、launch_source、completed_loops UI 进度、来源和设备状态展示
ScenarioSpec.timeline_items() 时间轴生成入口 按 ts_ns 把帧和链路动作统一排序 决定调度主顺序
推论：同时间戳的先后顺序 当前 timeline_items() 只按 ts_ns 排序，没有定义显式次级优先级。 因为 Python 排序是稳定的，而列表构造顺序是“帧先加入、链路动作后加入”，所以同一时间戳下当前行为会表现为“帧先于链路动作”。

调度内核

引擎真正的时序决策都集中在 engine.py:251 的 _run_loop()。

核心循环
两个关键常量
  • FRAME_SLICE_WINDOW_NS = 2_000_000：连续帧的批处理窗口是 2ms。
  • FRAME_QUEUE_LOOKAHEAD_NS = 20_000_000：未来 20ms 内可以考虑提前交给适配器排队。
统一时间基准
  • 启动时记录 _base_perf_ns = time.perf_counter_ns()。
  • 每个时间轴项的目标时刻是 target_ns = _base_perf_ns + item.ts_ns。
  • 暂停恢复时平移的是 _base_perf_ns，不是改时间轴。
while True:
    if stop_requested:
        return

    if state == PAUSED:
        wait(0.1s)
        continue

    if timeline_index >= len(timeline):
        if not handle_timeline_exhausted():
            return
        continue

    frame_batch = frame_batch_at(timeline_index)
    item = frame_batch[0] if frame_batch else timeline[timeline_index]
    advance_count = len(frame_batch) if frame_batch else 1

    update_runtime_snapshot_for_item(item, timeline_index)

    target_ns = base_perf_ns + item.ts_ns
    now_ns = perf_counter_ns()

    if frame_batch and should_schedule_frame_batch(frame_batch, target_ns, now_ns):
        dispatch_frame_batch(frame_batch, scheduled=True)
        timeline_index += advance_count
        continue

    if now_ns < target_ns:
        sleep_until(target_ns)
        continue

    if frame_batch:
        dispatch_frame_batch(frame_batch)
    else:
        dispatch(item)

    timeline_index += advance_count
三条核心分支 当前索引项只会进入三条路径之一：
1. 未来且满足条件时，整批帧提前走计划发送；
2. 还没到目标时刻时，线程继续等待；
3. 已到点或已超时时，立即发送或立即执行链路动作。
计划发送一旦成立，索引会立即前进 这意味着引擎线程自己不会陪着这个批次等到目标时刻，而是把“精准触发”的责任交给适配器的 send_scheduled(batch, enqueue_base_ns)，随后继续向后看时间轴。

帧批处理与发送

批处理既看时间，也看时间轴连续性；链路动作会成为天然屏障。

batching
批次如何形成
  1. 只有当前索引是 FrameEvent 才尝试并批。
  2. 从首帧开始向后扫，直到遇到非帧事件或超出 2ms 窗口。
  3. 条件是 item.ts_ns < first + 2ms，所以“恰好 2ms”不算同一批。
  4. 只要中间插入一个 LinkAction，批次立刻被切断。
发送前还要再处理一轮
  1. 先按 FrameEnableService 过滤禁用帧。
  2. 对保留帧做 SignalOverrideService.apply()。
  3. 把逻辑通道改写成物理通道。
  4. 再按适配器分组，逐组调用 send() 或 send_scheduled()。
阶段 方法 当前行为 调度意义
分片 _frame_batch_at() 2ms 内、连续出现、且全是帧，才会形成批次 决定“要不要一起发”
过滤 frame_enables.is_enabled() 被禁用的帧不发送，直接计入 skipped_frames 批次大小会动态缩小
覆盖 signal_overrides.apply() 在逻辑通道维度做 DBC/J1939 编码补丁 最终发送 payload 和日志 payload 都会改变
映射 frame.clone(channel=physical_channel) 把逻辑通道号改成实际设备使用的物理通道号 引擎调度用逻辑通道，适配器发送用物理通道
归组 frames_by_adapter.setdefault(...).append(...) 同一适配器保持原顺序，不同适配器拆开发送 逻辑同批次在设备侧可能是多个 adapter batch
统计 _send_prepared_frames() 支持部分发送成功；未发出的部分计入跳过并打 warning 驱动拒收或部分接收会体现在统计和日志里
推论：同适配器多通道的顺序 当前代码是在时间轴顺序上边遍历边 append，所以同一适配器下、不同逻辑通道但属于同一批次的帧，会保持原始时间顺序； 现有测试也验证了这一点。
计划发送条件 当前实现 结果
至少存在启用帧 enabled_frames = _enabled_frames(frames) 空批次直接放弃计划发送
目标时刻在未来 target_ns > now_ns 已经到点或过点时改走立即发送
未来不超过 20ms target_ns - now_ns <= 20ms 太远的批次不提前排队
总线类型可排队 SCHEDULED_FRAME_BUS_TYPES = {CAN, J1939} 启用帧里只要出现 CANFD，就整批回退到立即发送
相关适配器都支持 queue_send adapter.capabilities().queue_send 任一适配器不支持则整批不排队
一个容易漏掉的细节 计划发送资格只看“启用帧”。所以如果同一批里有一帧 CANFD，但它已经被帧使能规则禁掉，那么剩下的可用 CAN/J1939 帧仍然可能走计划发送。

暂停、停止与循环

这部分决定了暂停后会不会补发过期帧、结束后保留什么状态、循环模式如何进入下一圈。

state machine
暂停 / 恢复
  • pause() 只做两件事：状态改为 PAUSED，记录 _pause_started_ns。
  • 调度线程在暂停态下每 0.1 秒醒一次，等待恢复或停止。
  • resume() 执行 _base_perf_ns += now - _pause_started_ns。
  • 这等价于把整个逻辑时间轴整体向后平移，从而避免恢复时把暂停期间错过的时刻全算成超时。
等待策略
  • _sleep_until() 最多一次睡 2ms，不会长时间闷睡。
  • 每轮醒来都会重新检查 stop 和 pause。
  • 这让用户的暂停/停止操作能在等待阶段尽快生效。
场景 关键方法 索引 / 时间基准处理 结果
手动停止 stop() 设置 _stop_requested，线程 join，timeline_index 重置为 0 状态变成 STOPPED，当前源文件清空
自然播放完成 _handle_timeline_exhausted() 不循环时 timeline_index = len(timeline) 快照保留最后处理到的时间和源文件
循环播放 _restart_loop_playback() 完成圈数 +1，重开适配器与通道，索引归零，重新记录 _base_perf_ns 继续保持 RUNNING 并进入下一圈
循环回放不是简单“把索引拨回 0” 当前实现会先执行 _teardown_adapters() 再执行 _prepare_channels()，所以跨圈的本质是“重新初始化通道后重播”。
运行态帧禁用规则要分两层理解 引擎层不会在跨圈时清空 FrameEnableService，因此“本次回放过程中临时禁掉某帧”会跨循环边界持续生效。
但应用层会在 start_replay()、stop_replay()，以及检测到引擎从非停止转为停止时清空它们， 让这些规则只活在当前一次回放会话里。

运行时可见性

UI 看到的进度、来源、设备状态和循环圈数，都来自引擎快照与应用层的状态清理。

snapshot
快照中的关键字段
  • state：STOPPED / RUNNING / PAUSED
  • current_ts_ns、total_ts_ns：驱动进度显示
  • timeline_index、timeline_size：表示调度当前位置
  • current_source_file：当前帧源文件名，或链路动作标记
  • adapter_health：适配器与每个物理通道的健康信息
  • launch_source、loop_enabled、completed_loops
UI 实际显示出的语义
  • 回放来源显示为“场景绑定文件”或“主窗口选中文件回退”。
  • 循环模式下统计文案会显示“当前第几圈 / 已完成几圈”。
  • 设备状态会按适配器聚合，并细到逻辑通道对应的物理通道。
  • 自然完成与手动停止的快照差异，能帮助 UI 区分“播完了”还是“被用户按停”。

当前行为清单

这是对“当前调度到底怎么工作”的高密度总结，适合后续改代码时快速对照。

behavior checklist
主题 当前行为 说明
trace 合并 启动前统一加载、统一排序 不是运行中逐个文件切换
同时间戳顺序 当前推论为“帧先于链路动作” 来自稳定排序和构造顺序
批处理边界 2ms 内、连续且全是帧才会并批 链路动作和 2ms 边界都会切断批次
计划发送 只看未来 20ms 内、且总线类型与适配器能力都满足条件的启用帧 当前 CANFD 不走计划发送
通道启动冲突 同一物理通道若存在不一致的启动配置，会在启动阶段直接报错 相同配置允许复用，不同配置拒绝启动
信号覆盖 发送前生效，并影响调试日志 覆盖发生在逻辑通道维度
帧禁用作用域 键是 (logical_channel, message_id) 相同 ID 在别的逻辑通道上不受影响
暂停恢复 恢复时平移时间基准，不补发暂停期间错过的帧 这是避免恢复瞬间刷出过期帧的关键
循环跨圈 会重开适配器和通道，再从索引 0 重播 不是单纯把索引改回 0
部分发送 成功部分计入 sent，剩余部分计入 skipped，并打 warning 日志只记录真正发出的那些帧
